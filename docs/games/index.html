Performance

Particle Storm

48k particles — Particle[] vs struct-of-arrays Float64Array; FPS and physics ms/frame.

vs TypeScript: fewer heap objects and sequential numeric memory usually cut physics time (often a clear gap on the same CPU) because the hot loop stops chasing object references and loads cache-friendly lanes.

 Performance

Grid Life

Conway on 180×200 — boolean[][] vs flat Uint8Array; step time and FPS.

vs TypeScript: one contiguous buffer vs many row arrays — neighbor reads become a tight stride through RAM, so step ms usually drops (less pointer chasing and better locality).

 Performance

Vector Bench

30k vectors — {x,y,z}[] vs packed Float64Array; batch normalize kernel.

vs TypeScript: array-of-structs scatters work across objects; packed Float64Array keeps each component adjacent so the inner normalize loop is friendlier to the CPU cache and JIT.

 Performance

Alloc Strike

65k-sample fill — new Float64Array every frame vs one reused buffer (GC / allocator).

vs TypeScript: per-frame allocation feeds GC and malloc; reuse removes that cost so frame time is steadier (often much lower ms and fewer hitches).

 Performance

Path BFS

Grid flood fill — Array.shift() FIFO vs Int32Array ring queue.

vs TypeScript: shift() rewrites the queue; a ring index on typed memory is O(1) at both ends, so BFS rounds usually show much lower compute ms.

 Performance

Count Sort Surge

100k keys — counting sort on number[] vs Uint32Array.

vs TypeScript: counting passes touch every element repeatedly — Uint32Array avoids boxed number elements so the inner loops stay dense and fast.

 Performance

Matrix Core

32×32 matmul — number[][] vs row-major Float64Array.

vs TypeScript: nested arrays add indirection per row; row-major packed storage matches how CPUs expect to stream floats, so multiply time usually improves.

 Performance

Bucket Downpour

24k bucket inserts — nested push vs count + flat scatter.

vs TypeScript: many small push chains mean more backing-store growth and pointer work; count-then-scatter into one slab reduces allocator traffic.

 Performance

Queue Blitz

FIFO stress — shift() vs power-of-two ring buffer.

vs TypeScript: this is a deliberate worst case for shift (linear copy each pop); the ring buffer stays O(1), so the ms gap is usually large.

 Performance

Spring Chain

3.8k masses — { x, v }[] vs two Float64Array lanes; spring forces + integration.

vs TypeScript: struct-of-arrays removes one heap node per mass; neighbor updates become index math on dense buffers, so integration usually wins on compute ms.

 Performance

Blur Pass

120×120 box blur — number[][] vs swapped Uint8Array planes.

vs TypeScript: stencil reads hit double indirection and boxed values on number[][]; flat bytes keep the kernel memory-bound but in a straight line through RAM.

 Performance

Binary Hunt

65k sorted keys, 20k lower_bound calls/frame — number[] vs Uint32Array.

vs TypeScript: same algorithm; typed slab uses fixed 4-byte elements without per-slot boxing, which often lowers probe cost when the table is large.
