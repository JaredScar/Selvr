Batch math: objects vs packed buffers

Both panels run the same workload on 30 000 3D vectors each frame: a small sinusoidal perturbation followed by normalize. TypeScript keeps one { x, y, z } object per vector; the Selvr-style path stores x,y,z contiguously in a single Float64Array — fewer indirections and better locality for numeric kernels.

The canvas shows a sparse polyline of projected x/y (cheap visual); the metric that matters is compute ms/frame.

Expect similar numbers on both sides. Here the loop is dominated by sin, cos, and sqrt, not by loading and storing vector components. Engines like V8 also optimize a uniform { x, y, z } array to near–fixed-offset access, so packed buffers do not always look faster in plain JS — the point is the same workload expressed the way a compiler can lower to tight buffers and non-JS targets.

Why this is fair

  • Same N, same perturbation + normalize math, same lightweight draw — only vector storage differs.
  • This complements Particle Storm (SoA for moving points) with a classic batch linear algebra shape.

Why speeds are often very similar

  • Math-bound work: Most time goes into transcendental and sqrt calls, so memory layout is a smaller slice of the total than in demos that are mostly simple arithmetic over huge data (e.g. Particle Storm or Grid Life).
  • JIT-friendly objects: A Vec3[] of identical shapes is not “slow generic objects” in modern engines — property access often compiles to something close to struct field access.
  • No per-frame allocation in the hot loop on either side, so this is not a GC comparison; Selvr’s packed-layout story matters more when you want predictable lowering to bytecode or WASM and when allocation would otherwise churn.
