Why TypeScript hits a wall here

Core Harvest is the same mechanic on both sides: start just below Number.MAX_SAFE_INTEGER and add +1 per harvest. In JavaScript, number is an IEEE-754 double — not every integer exists above 253 − 1, so increments stop changing the stored value while the true count keeps going. TypeScript does not fix this: types are erased, and number is still a double at runtime. bigint exists, but mixing it with number, JSON, and most APIs is awkward; games often still use number and silently break.

Selvr is aimed at sound integer semantics (e.g. i64) for this kind of logic — exact arithmetic without floating-point surprises, aligned with the bytecode VM story in the README.

What to watch

  • TypeScript: after a couple of harvests, the big number on top stops increasing while “Correct total (BigInt)” keeps going — that gap is real precision loss, not a UI bug.
  • Selvr: mass uses BigInt here to mirror exact integer codegen; every +1 is exact.

Summary

TypeScriptSelvr (design target)
Large integers number is a double; unsafe above 253−1. bigint is opt-in and clunky at boundaries. Fixed-width integers (i64, etc.) for exact economy / scores / seeds.
Types help? Compile-time only; number is still a double after emit. Sound types through the pipeline toward VM / bytecode.
