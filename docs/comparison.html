TypeScript vs Selvr

Five representative tasks, shown side by side. Selvr syntax is intentionally close to TypeScript — existing TypeScript developers can read it immediately. The differences are deliberate upgrades.

1
Functions & Types
How each language defines typed functions and variables
TypeScript
// TypeScript — `number` is the only numeric type
// No distinction between int and float

function add(a: number, b: number): number {
    return a + b;
}

// Type `any` is always a valid escape hatch
function doAnything(x: any): any {
    return x.whatever.doesnt.exist; // compiles fine!
}

const x: number = 42;
let y = 3.14; // same type as x

// Implicit coercions compile silently:
const total = x + "5"; // "425" — surprise!
Selvr
// Selvr — precise numeric types catch bugs early
// `number` alias exists for TypeScript familiarity

fn add(a: i32, b: i32): i32 {
    return a + b;
}

// `any` does not exist — every value has a known type
// fn broken(x: any) — COMPILE ERROR: `any` is not a type

const x: i32  = 42;
const y: f64  = 3.14; // different type to x

// Implicit coercions are compile errors:
// const total = x + "5"; — ERROR: cannot add i32 and string
const total = x + parseInt("5").unwrapOr(0);
✓
Same colon syntax TypeScript developers can read Selvr function signatures immediately. The only difference is fn instead of function.
✓
Precise numeric types i32, i64, f32, f64 — no silent integer overflow or float precision surprises. The number alias is available for teams migrating from TypeScript.
✗ TS
TypeScript's any is a full type-safety exit hatch. In large codebases it spreads silently and can disable the type checker for whole call chains. Selvr has no any.
2
Null Safety
How each language handles absent values
TypeScript
// TypeScript — optional types via T | undefined | null
// Requires --strictNullChecks to be useful at all

interface User {
    name: string;
    email?: string; // string | undefined
}

function getEmail(u: User): string | undefined {
    return u.email;
}

// Easy to forget the check:
const upper = getEmail(user).toUpperCase(); // runtime crash!

// Correct, but verbose:
const safe = getEmail(user)?.toUpperCase() ?? "(none)";

// TypeScript still allows:
const bad: string = null as any; // compiles!
Selvr
// Selvr — no null, no undefined, no exceptions
// Absence is a typed value: Option<T>

struct User {
    name: string;
    email: Option<string>; // explicit, always
}

fn getEmail(u: User): Option<string> {
    return u.email;
}

// The compiler forces you to handle absence:
// const upper = getEmail(user).toUpperCase(); — COMPILE ERROR
// Option<string> has no method `toUpperCase`

// Clean, enforced pattern:
const safe = getEmail(user)
    .map((e) => e.toUpperCase())
    .unwrapOr("(none)");
✓
Option<T> is explicit and enforced at compile time. There is no way to call a method on an absent value — the compiler rejects it. TypeScript's T | undefined requires --strictNullChecks and still allows any-based bypasses.
✓
No null or undefined in the language at all. These concepts simply don't exist. The billion-dollar mistake is not possible.
ℹ
Option is an enum, not special syntax. map, andThen, unwrapOr, and ? work on any Option<T>. TypeScript developers familiar with functional patterns will feel at home immediately.
3
Error Handling
Typed errors vs. thrown exceptions
TypeScript
// TypeScript — errors are thrown (invisible in types)
// The caller has no idea this function can fail

async function fetchUser(id: number): Promise<User> {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error("Network error");
    return res.json(); // can also throw!
}

// All error paths are invisible:
const user = await fetchUser(1); // might throw, nothing warns you

// To handle safely, must know to wrap in try/catch:
try {
    const user = await fetchUser(1);
    console.log(user.name);
} catch (e: unknown) {
    console.error("failed"); // e has type `unknown`
}
Selvr
// Selvr — errors are values, visible in the return type
// The type signature tells you everything that can go wrong

async fn fetchUser(id: i32): Result<User, FetchError> {
    const res = fetch(`/api/users/${id}`).await?; // ? = propagate on err
    const user = res.json<User>().await?;
    return Ok(user);
}

// Compiler REQUIRES you to handle both cases:
match fetchUser(1).await {
    Ok(user)    => console.log(user.name),
    Err(FetchError.NotFound) => console.log("not found"),
    Err(e)      => console.log(`error: ${e}`),
}

// Or propagate with ? — readable error chains
const user = fetchUser(1).await?;
✗ TS
TypeScript's exception model is invisible. A function that throws Error looks identical to one that doesn't. There is no way to know which functions can fail without reading their implementation.
✓
Result<T, E> makes failure explicit in the type. You can see from the signature exactly what can go wrong, and the compiler forces you to handle it — or explicitly propagate with ?.
ℹ
The ? operator is zero-cost syntactic sugar. value? desugars to if Err(e), return Err(e); else unwrap Ok(v) — identical to what you'd write manually, but concise.
4
Pattern Matching
Exhaustive branching on data shapes
TypeScript
// TypeScript — discriminated unions + switch

type Shape =
  | { kind: "circle";    radius: number }
  | { kind: "rectangle"; width: number; height: number }
  | { kind: "triangle";  base: number;  height: number };

function area(s: Shape): number {
  switch (s.kind) {
    case "circle":    return Math.PI * s.radius ** 2;
    case "rectangle": return s.width * s.height;
    case "triangle":  return 0.5 * s.base * s.height;
    // Forgetting a case is NOT a compile error
    // unless you use a never-type trick
  }
  return 0; // fallback required
}
Selvr
// Selvr — algebraic enum + exhaustive match

enum Shape {
    Circle    { radius: f64; },
    Rectangle { width: f64; height: f64; },
    Triangle  { base: f64;  height: f64; },
}

fn area(s: Shape): f64 {
    return match s {
        Shape.Circle    { radius }        => Math.PI * radius * radius,
        Shape.Rectangle { width, height } => width * height,
        Shape.Triangle  { base, height }  => 0.5 * base * height,
        // Removing any arm = COMPILE ERROR: non-exhaustive match
    };
}

// Guards, ranges, or-patterns — all in one expression
const label = match score {
    90..=100 => "A",
    80..=89  => "B",
    x if x < 0 => "invalid",
    _ => "C or below",
};
✓
match is exhaustive by default. Adding a new variant to an enum causes all existing match expressions to become compile errors — you can never forget to handle a new case.
✓
match is an expression, not a statement. It returns a value — no need for let result; switch (...) { case ...: result = ...; } patterns.
△
TypeScript requires a never-based trick for exhaustiveness. const _: never = s; in the default branch. It works, but it's not built in to switch itself.
5
Async & Fetch
Async/await and typed network requests
TypeScript
interface Post { id: number; title: string; body: string; }

async function loadPost(id: number): Promise<Post> {
    const res  = await fetch(`/api/posts/${id}`);
    const post = await res.json() as Post; // unsafe cast!
    return post;
}

// No forced error handling — this compiles without try/catch
const post = await loadPost(1);
console.log(post.title); // crashes if fetch fails

// Concurrent requests via Promise.all
const [a, b] = await Promise.all([
    loadPost(1),
    loadPost(2),
]);
Selvr
struct Post { id: i32; title: string; body: string; }

async fn loadPost(id: i32): Result<Post, FetchError> {
    const res  = fetch(`/api/posts/${id}`).await?; // ? propagates
    const post = res.json<Post>().await?; // type-safe deserialize
    return Ok(post);
}

// Compiler REQUIRES handling the error case
match loadPost(1).await {
    Ok(post) => console.log(post.title),
    Err(e)   => console.log(`Error: ${e}`),
}

// Concurrent — same semantics as Promise.all
const [a, b] = join!(loadPost(1), loadPost(2)).await;
✗ TS
res.json() as Post is an unchecked cast. TypeScript trusts you to get the shape right. If the API returns a different shape at runtime, you get silent undefined fields — no compile error, no runtime error until you access the field.
✓
Selvr's json<T>() validates at the boundary. If the JSON doesn't match the struct, you get an Err — not a silent crash later in the program.
✓
Async/await syntax is identical. If you know async/await in TypeScript, you know it in Selvr. The only difference is .await (postfix) instead of await expr (prefix) — which composes more naturally in chains.
Feature comparison at a glance
How the two languages handle the same problems
Feature TypeScript Selvr
Null safety △ Optional with --strictNullChecks, bypassed by any ✓ No null or undefined in the language
Error handling △ Thrown exceptions — invisible in types ✓ Result<T, E> — errors visible in every signature
Exhaustive matching △ Requires never-type trick in switch ✓ match is exhaustive by default
Type escape hatch ✗ any, as unknown as T ✓ No any; as only valid between compatible types
Memory management △ Garbage collected — unpredictable GC pauses ✓ Ownership model — deterministic, GC-free
Familiar syntax ✓ Baseline ✓ Identical to TypeScript in most places
Cold start speed △ JS parse tax on every cold load ✓ Bytecode — no parser, ~8–12× faster cold start
Runtime performance △ JIT-optimised, but GC pauses cause jank ✓ Compiled to native bytecode, GC-pause-free
Generics ✓ Generic types with <T> ✓ Same <T> syntax, monomorphised at compile time
Pattern matching ✗ No native match ✓ First-class match expression with guards
