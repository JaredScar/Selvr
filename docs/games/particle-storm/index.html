Performance: data layout, not syntax sugar

Both panels run the same physics on 48 000 particles and the same sparse draw (every 24th point). The difference is memory layout: the TypeScript side uses a heap object per particle (a very common pattern); the Selvr-style side uses struct-of-arrays (Float64Array blocks) with no allocations in the animation loop — what the Selvr compiler is aimed at emitting for hot simulation code.

Watch FPS and physics ms/frame: the SoA path should stay higher / lower respectively on typical hardware. Phase 2 bytecode / WASM targets the same tight loops without shipping a giant parseable JS surface to the browser.

Why this is fair

  • Same N, same integration, same canvas size, same draw stride — only the representation of particles differs.
  • TypeScript is not “wrong” — many codebases choose objects for clarity. Selvr’s pitch is that the compiler can lower structured code to fast buffers so you don’t hand-roll SoA.
  • This demo runs in the browser as JS today; Selvr’s bytecode VM is the next step for skipping parse/JIT on that hot path.
