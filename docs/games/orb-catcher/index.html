Why Selvr for browser games

TypeScript is excellent for authoring, but types are erased when you ship. At runtime the browser still runs plain JavaScript, with the same legacy semantics and a full GC. Selvr is aimed at a different end state: a sound type system that stays meaningful in the compiled artifact, deterministic memory for hot paths, and (in Phase 2) bytecode so the browser does less parse and JIT work.

Topic TypeScript (this demo) Selvr
Types at runtime Stripped; as casts can hide real bugs. Designed for sound types through the pipeline; no typeof null === "object" legacy.
Shipping Source + build step; runtime is still JS semantics + GC. Compiler-first; tree-shaken JS today, bytecode VM path for cold start and predictable performance.
DOM / canvas Often optional chaining + assertions for canvas/context. Option-style APIs (.unwrap() in source) encourage explicit handling instead of silent undefined.
Frame pacing GC and JS engine heuristics can add jitter. Ownership model avoids GC pauses in engine code (see project roadmap).
